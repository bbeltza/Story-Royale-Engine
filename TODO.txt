- Coroutines (Cooperative multitasking):
	- Cooperative multitasking is a way of multitasking (doing multiple operations simultaneously)
	  in which YOU decide when to switch to the next task by calling a special function without finishing the function.
	- You will be able to for example wait for V seconds and not block anything else
	- It's like multithreading but usually more synchronized, tasks will run on one thread (separate from the main one) or perhaps on a thread pool.
	- But if you do the infamous "while (true)" non exiting loop then you'll block the thread and thus there won't be any coroutines that can run
	  (Well unless I go with the thread pool in which there will be less threads left and the blocking thread will burn the entire CPU)
	- Might also be a Core feature, or from Base if it can be run without any of the engine

	- The reason why I want this: It's messy to be creating multiple threads in my game to do one thing like transitioning from one scene to another
								  and thus having to figure out how to avoid the thread without crashing when the game closes because it's using a pointer
								  to X that has already been cleaned up
	- I did not add this before because it's so complex, context switching is literally implemented at the assembly level so I have to do this for every
	  architecture and OS that I have to support. Win32 has Fibers, they do what I want, and that's what I got inspired of to have this idea. So implementing
	  coroutines on Windows would be pretty easy, I saw some of the desassembly of the Fiber API and yet it has to save the state of every register and there's
	  no quick system/Nt call for that (It only uses them to setup the stacks), what a silly thing to make for every architecture!
	  I saw that linux has something about contexts, I still don't know exactly how to use them so I still need to see

		- On Windows, you can have the game as a separate process executable, with its own memory space and have the engine create the process when running the game:
			- This allows to garantee the engine's initialization when initializing game global variables, so it solves most static initialization and destruction overheads.
			- It also disallows the user to touch any internal engine variables. That's what most game engine's approach is.
			- To still have the game run on a single executable, I can use the build system to build the game executable and store it inside an array of raw bytes.
				- However, as you cannot create processes from a data stream, I'll have to create a temporal file and write the raw data inside, and load that file with the OS api.
			- The "game executable" can also be a DLL and I can still load it having its own memory space.
		- However there are some difficulties:
			- I'll have to find a way to call the engine functions used by the games to work with
			- I'll also have to implement this for every single operating system, as it is very OS specific. However, only 2 major platforms are mostly important: Windows and POSIX (Unix like operating systems including linux, android if it works, and FreeBSD (with inherits macOS, PlayStation and many others))
